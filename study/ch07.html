CHAPTER 07

工作流与实验管理 Workflow

前六章我们把数据、模型、策略、回测这些"零件"逐一拆开讲透了。本章要做的事只有一件：把这些零件用一根传送带串起来，一键跑完，并且把每次实验的结果都规规矩矩地存档、随时能翻出来复现。

1. 为什么需要"工作流"？一条流水线的故事

想象你在一家工厂造车。前面几章你分别学会了：怎么进原料（数据层）、怎么把原料加工成零件（数据集与处理器）、怎么组装发动机（模型）、怎么装上轮子跑起来（策略与回测）。但如果每造一辆车都要人工把这些工序手动搬来搬去，效率极低，而且很容易漏工序、记错参数。

真正的工厂需要两样东西：一条 自动流水线（工序按顺序自动流转），和一本 生产台账（每辆车用了什么参数、质检结果如何，都记录在案，出了问题能追溯）。Qlib 的 Workflow（工作流） 就是这条流水线，Recorder（记录器） 就是这本台账。

一句话概括 Workflow 负责"按顺序自动跑完 数据 → 训练 → 预测 → 信号分析 → 组合回测"；实验管理（Recorder / ExpManager）负责"把每一步的参数、模型、结果、图表都存起来，可对比、可复现"。

Qlib 提供 两种 使用方式，本章都会讲：

  方式 A：命令行驱动（配置即实验）
  ┌───────────────────────────────────────────────┐
  │  $ qrun workflow_config_lightgbm_Alpha158.yaml │
  └───────────────────────────────────────────────┘
        一个 YAML 文件描述整条流水线，qrun 一键跑完

  方式 B：代码驱动（灵活可编程）
  ┌───────────────────────────────────────────────┐
  │  with R.start(experiment_name="my_exp"):       │
  │      model.fit(dataset)                         │
  │      R.save_objects(...)                        │
  │      SignalRecord(...).generate()               │
  └───────────────────────────────────────────────┘
        用 Python 逐步调用，适合调试与二次开发

两种方式底层是同一套东西：qrun 内部最终也是调用代码驱动的那套 API。理解了代码驱动，就理解了 qrun。

2. qrun 命令行入口：一个 YAML 跑完全流程

安装好 Qlib 后你就有了 qrun 命令，它的入口注册在 setup.py 里，指向函数 qlib.cli.run:run。整段逻辑非常短，就在 qlib/cli/run.py:86 的 workflow() 函数里。它借助 fire 库把命令行参数映射成函数参数，用法是：

# 最简单：跑一个配置
$ qrun workflow_config_lightgbm_Alpha158.yaml

# 也可以指定实验名和 mlruns 目录
$ qrun config.yaml my_experiment ./mlruns

它的执行步骤（对照 qlib/cli/run.py:86-148）：

  1. Jinja2 模板渲染：先把 YAML 当模板读进来，用环境变量做替换（render_template，run.py:52）。所以你可以在 YAML 里写 {{ MARKET }}，然后 export MARKET=csi300 再跑，同一个配置适配不同环境。
  2. 解析 YAML：用 ruamel.yaml 把渲染后的文本解析成 Python 字典。
  3. 处理 BASE_CONFIG_PATH：若配置里写了 BASE_CONFIG_PATH，会先加载基础配置，再用当前文件里的字段覆盖（update_config），实现"配置继承"。
  4. 初始化 Qlib：调用 qlib.init(**config["qlib_init"])，同时设置实验管理后端的 uri（默认落到当前目录的 mlruns/）。
  5. 训练与记录：核心一行 recorder = task_train(config["task"], experiment_name=...)，把 task 段交给训练器执行；最后 recorder.save_objects(config=config) 把整份配置也一并存进 recorder，保证可复现。
提示 task_train（在 qlib/model/trainer.py）才是真正把 model / dataset / record 三段拼起来的地方：它先构建 dataset 和 model，训练模型，然后依次实例化 record 段里列出的每个 RecordTemp 并调用 .generate()。qrun 只是它的一层薄壳。
易踩坑 Jinja2 渲染发生在 YAML 解析之前。如果你在 YAML 里写了 {{ ... }} 但没设置对应环境变量，该变量会被替换成空字符串（见 run.py:77 的 os.getenv(var, "")），可能导致后续解析出意料之外的 None。

3. 全局记录器 R：实验管理的总开关

代码驱动的核心是一个全局单例对象 R，定义在 qlib/workflow/__init__.py:681：

from qlib.workflow import R

你可以把 R 想象成台账管理员：它管着"哪个实验（Experiment）正在进行"、"当前用哪本记录本（Recorder）在记录"。它本质是 QlibRecorder 类的实例，内部持有一个 ExpManager（实验管理器）。

层级关系要先分清（这是最容易混淆的地方）：

  ExpManager  实验管理器（单例，管所有实验，对应 mlruns/ 根目录）
     │
     ├── Experiment  一次"课题"，有名字，如 "LightGBM_Alpha158"
     │      │
     │      ├── Recorder  一次"运行/试验"，有唯一 id，记录一组参数与结果
     │      ├── Recorder
     │      └── Recorder  （同一实验下可以有很多次 run，方便对比）
     │
     └── Experiment ...

最常用的入口是 R.start(...) 上下文管理器（__init__.py:37）。它是一个 @contextmanager，进入 with 块时创建/激活一个 experiment 和 recorder，退出时自动收尾——正常结束标记为 FINISHED，抛异常则标记 FAILED：

from qlib.workflow import R

with R.start(experiment_name="train_lgb", recorder_name="run_1"):
    # 1) 记录超参数（键值对）
    R.log_params(learning_rate=0.2, max_depth=8)

    # 2) 训练
    model.fit(dataset)

    # 3) 记录指标（可带 step，用于画曲线）
    R.log_metrics(train_loss=0.33, step=1)

    # 4) 把 Python 对象（模型/预测）直接存为产物
    R.save_objects(trained_model=model)

    # 5) 拿到当前 recorder，记下它的 id 以便日后加载
    rid = R.get_recorder().id

几个核心方法速查（都在 QlibRecorder 上，见 __init__.py）：

  • R.log_params(**kw)（:542）：记录超参数，值不可变，一次实验记一次。
  • R.log_metrics(step=None, **kw)（:567）：记录评估指标，可随 step 多次记录形成曲线。
  • R.save_objects(**kw)（:481）：把 Python 对象（如 {"pred.pkl": pred}）直接序列化存为 artifact——这是比 MLflow 原生 log_artifact 更方便的封装。
  • R.load_object(name)（:536）：从当前 recorder 加载之前存的对象。
  • R.get_recorder(recorder_id=..., experiment_name=...)（:392）：按 id/名字取回任意 recorder，是"翻旧账"的关键。
设计动机 文件顶部的注释（__init__.py:3-15）解释了为什么不直接用 MLflow：Qlib 用 recorder 对象（有一堆直观方法）代替了到处传 run_id；提供了直接存/取 Python 对象的 save_objects/load_object；启动时还会自动记录代码 diff。代价是"必须先有 experiment 才能拿 recorder"。

4. 底层是 MLflow：ExpManager 与 mlruns 目录

R 只是门面，真正干活的后端是 MLflowExpManager（qlib/workflow/expm.py:317，继承自 ExpManager）。它把 Qlib 的实验/记录概念映射到 MLflow 的 experiment/run 上，并把所有数据落盘到一个 uri 指向的目录，默认是当前工作目录下的 mlruns/。

对应地，具体的记录本实现是 MLflowRecorder（qlib/workflow/recorder.py:247）。它的 save_objects（:397）用 pickle 序列化对象、load_object（:413）反序列化、log_params/log_metrics 转调 MLflow 的对应 API。

安全提醒 load_object 走的是 pickle 反序列化。不要加载来源不明的 recorder 产物，pickle 可执行任意代码。Qlib 近期已在部分加载路径引入 RestrictedUnpickler 收紧安全。

mlruns 目录长什么样？跑完一次 qrun 后，你会在当前目录看到类似结构：

  mlruns/
   └── 1/                         <- experiment_id（实验）
        ├── meta.yaml             <- 实验元信息（名字等）
        └── a1b2c3.../            <- recorder_id（一次 run）
             ├── meta.yaml        <- run 的元信息、状态
             ├── params/          <- log_params 记的超参数
             ├── metrics/         <- log_metrics 记的指标（IC/ICIR...）
             ├── tags/            <- set_tags 记的标签
             └── artifacts/       <- save_objects 存的对象
                  ├── pred.pkl              (SignalRecord)
                  ├── label.pkl             (SignalRecord)
                  ├── sig_analysis/         (SigAnaRecord)
                  └── portfolio_analysis/   (PortAnaRecord)

你可以用 MLflow 自带的 UI 可视化对比这些实验：

# 在含有 mlruns/ 的目录下执行
$ mlflow ui
# 然后浏览器打开 http://127.0.0.1:5000 对比各次 run 的指标

5. 记录模板 RecordTemp：可复用的流水线步骤

如果每个人都要自己写"预测 → 算 IC → 回测"的代码，既重复又容易出错。Qlib 把这些常见步骤抽象成 记录模板（RecordTemp），定义在 qlib/workflow/record_temp.py。它们都接收一个 recorder，调用 .generate() 就会自动算好结果并存进 recorder。三个最重要的模板形成一条依赖链：

  SignalRecord  →  SigAnaRecord  →  PortAnaRecord
   生成预测          分析信号质量        组合回测分析
  (pred/label)      (IC/ICIR)         (收益/风险)

  后两者的 depend_cls = SignalRecord：
  它们依赖前者产出的 pred.pkl / label.pkl 才能工作

① SignalRecord（record_temp.py:161）——生成预测信号。

# 核心逻辑（record_temp.py:190）
def generate(self, **kwargs):
    pred = self.model.predict(self.dataset)   # 用模型对 test 段打分
    if isinstance(pred, pd.Series):
        pred = pred.to_frame("score")
    self.save(**{"pred.pkl": pred})     # 存预测
    raw_label = self.generate_label(self.dataset)
    self.save(**{"label.pkl": raw_label})  # 存真实标签

它产出 pred.pkl（模型预测分数）和 label.pkl（真实收益标签），是后续一切分析的基础。

② SigAnaRecord（record_temp.py:295）——信号分析。它加载 pred.pkl 与 label.pkl，计算 预测能力指标：

# 核心逻辑（record_temp.py:323）
ic, ric = calc_ic(pred.iloc[:, 0], label.iloc[:, self.label_col])
metrics = {
    "IC": ic.mean(),                 # 信息系数：预测与真实收益的相关性
    "ICIR": ic.mean() / ic.std(),     # IC 的稳定性（越高越稳）
    "Rank IC": ric.mean(),
    "Rank ICIR": ric.mean() / ric.std(),
}
self.recorder.log_metrics(**metrics)     # 指标写进 recorder

IC/ICIR 是量化选股里衡量"因子/模型预测好不好"的黄金指标。若开启 ana_long_short=True，还会额外算多空组合的年化收益与夏普。

③ PortAnaRecord（record_temp.py:358）——组合回测分析。它拿预测信号，套上第 6 章讲的策略（如 TopkDropoutStrategy）跑回测，产出：

  • report_normal.pkl：逐日收益报告；positions_normal.pkl：每日持仓明细。
  • port_analysis.pkl：组合的风险分析（年化收益、波动、夏普、最大回撤等，由 risk_analysis 计算）。
自动检查机制 SigAnaRecord 和 PortAnaRecord 都继承 ACRecordTemp（record_temp.py:212），它的 generate() 会先 check() 依赖是否就绪（pred.pkl 是否存在），缺依赖就跳过并告警，而不是直接报错崩溃。设 skip_existing=True 还能跳过已生成的结果，避免重复计算。

6. 逐段读懂一个 workflow YAML

把前面所有概念拼起来，就是一个 workflow 配置文件。我们逐段解剖 examples/benchmarks/LightGBM/workflow_config_lightgbm_Alpha158.yaml。它分四大块：全局初始化、市场/基准、数据配置、任务（task）。

① qlib_init + 市场/基准——初始化和一些用锚点复用的变量。YAML 里的 &market 是"定义锚点"，*market 是"引用锚点"，这样同一个值只写一次：

qlib_init:
    provider_uri: "~/.qlib/qlib_data/cn_data"  # 数据存放目录
    region: cn                              # A 股
market: &market csi300          # 定义锚点 market = 沪深300成分股
benchmark: &benchmark SH000300  # 定义锚点 benchmark = 沪深300指数

② data_handler_config——喂给数据处理器（第 4 章的 Handler）的参数，也定义成锚点，稍后被 dataset 引用：

data_handler_config: &data_handler_config
    start_time: 2008-01-01
    end_time: 2020-08-01
    fit_start_time: 2008-01-01     # 拟合归一化统计量的起止（防止未来信息泄露）
    fit_end_time: 2014-12-31
    instruments: *market           # 引用上面的 market 锚点

③ port_analysis_config——给 PortAnaRecord 用的回测/策略配置。注意 signal: <PRED> 这个占位符，运行时会被替换成模型的真实预测：

port_analysis_config: &port_analysis_config
    strategy:
        class: TopkDropoutStrategy       # 第6章的 Topk-Drop 策略
        module_path: qlib.contrib.strategy
        kwargs:
            signal: <PRED>               # 占位符：运行时填入模型预测
            topk: 50                     # 持有分数最高的 50 只
            n_drop: 5                    # 每日换出 5 只
    backtest:
        start_time: 2017-01-01
        end_time: 2020-08-01
        account: 100000000               # 初始资金 1 亿
        benchmark: *benchmark
        exchange_kwargs:
            limit_threshold: 0.095       # 涨跌停限制
            deal_price: close            # 以收盘价成交
            open_cost: 0.0005            # 买入手续费
            close_cost: 0.0015           # 卖出手续费
            min_cost: 5

④ task——整条流水线的心脏，分 model / dataset / record 三小节。Qlib 用 class + module_path + kwargs 的"动态实例化"模式，把配置直接映射成 Python 对象：

task:
    model:                               # 模型：LightGBM
        class: LGBModel
        module_path: qlib.contrib.model.gbdt
        kwargs: { loss: mse, learning_rate: 0.2, max_depth: 8, ... }
    dataset:                             # 数据集：DatasetH + Alpha158 因子
        class: DatasetH
        module_path: qlib.data.dataset
        kwargs:
            handler:
                class: Alpha158
                module_path: qlib.contrib.data.handler
                kwargs: *data_handler_config   # 引用②的锚点
            segments:                    # 训练/验证/测试三段时间切分
                train: [2008-01-01, 2014-12-31]
                valid: [2015-01-01, 2016-12-31]
                test:  [2017-01-01, 2020-08-01]
    record:                              # 记录步骤列表：依次执行
        - class: SignalRecord            # ① 生成预测
          module_path: qlib.workflow.record_temp
          kwargs: { model: <MODEL>, dataset: <DATASET> }
        - class: SigAnaRecord            # ② 信号分析 IC/ICIR
          module_path: qlib.workflow.record_temp
          kwargs: { ana_long_short: False, ann_scaler: 252 }
        - class: PortAnaRecord           # ③ 组合回测分析
          module_path: qlib.workflow.record_temp
          kwargs: { config: *port_analysis_config }
占位符替换 <MODEL>、<DATASET>、<PRED> 是特殊占位符。task_train 会先构建好 model 与 dataset，再把它们填进 record 段的 <MODEL>/<DATASET>；预测生成后把 <PRED> 填进策略的 signal。record 是个有序列表，因此依赖顺序天然满足：先 Signal，再分析。

7. 代码驱动整条流水线 & 翻回历史结果

把上面 YAML 干的事用 Python 手写一遍，你就彻底理解了工作流。下面是等价的代码驱动版本：

import qlib
from qlib.workflow import R
from qlib.workflow.record_temp import SignalRecord, SigAnaRecord, PortAnaRecord
from qlib.utils import init_instance_by_config

qlib.init(provider_uri="~/.qlib/qlib_data/cn_data", region="cn")

# 用配置字典动态构建 model 与 dataset（与 YAML 一一对应）
model   = init_instance_by_config(model_config)
dataset = init_instance_by_config(dataset_config)

with R.start(experiment_name="lgb_alpha158"):
    # 1) 训练 + 记录模型
    model.fit(dataset)
    R.save_objects(trained_model=model)
    recorder = R.get_recorder()          # 当前活跃的记录本

    # 2) 依次跑三个记录模板（顺序 = 依赖顺序）
    SignalRecord(model, dataset, recorder).generate()
    SigAnaRecord(recorder).generate()
    PortAnaRecord(recorder, config=port_analysis_config).generate()

    rid = recorder.id                    # 记住 id 以便日后翻账
    print("recorder id =", rid)

如何加载回历史结果？实验跑完后，产物都在 mlruns/ 里。用 R.get_recorder 按 id 或名字取回记录本，再 load_object 拿产物做二次分析（画图、对比、再回测）：

# 关掉 Python 重新进来也没关系，只要 qlib.init 指向同一个 mlruns
qlib.init(provider_uri="~/.qlib/qlib_data/cn_data", region="cn")

# 按 id + 实验名取回历史记录本
recorder = R.get_recorder(recorder_id=rid, experiment_name="lgb_alpha158")

# 加载之前存的对象（注意带上 artifact 子目录前缀）
pred  = recorder.load_object("pred.pkl")
ic    = recorder.load_object("sig_analysis/ic.pkl")
report = recorder.load_object("portfolio_analysis/report_normal_1day.pkl")

# 列出该实验下所有 run，或用 DataFrame 做指标对比
recorders = R.list_recorders(experiment_name="lgb_alpha158")
df = R.search_records([recorder.experiment_id], order_by=["metrics.IC DESC"])
工程建议 调试阶段用代码驱动逐步跑（能 import pdb 打断点）；实验定型后写成 YAML 用 qrun 跑，配置即文档，团队协作和批量实验都更整洁。两者可无缝切换，因为共用同一套 task_train 与 Recorder。

📝 课后练习题

1. （概念）请说出 ExpManager、Experiment、Recorder 三者的层级关系，并说明 R.get_recorder(recorder_id="xxx") 在没有 active recorder 时为什么必须同时提供 experiment_name？
参考答案
层级为 ExpManager（单例，管所有实验）→ Experiment（一个课题，有名字）→ Recorder（一次 run，有唯一 id）。见正文架构图与 expm.py。当没有 active recorder 时，系统无法知道该去哪个实验下按 id 找记录本，所以必须指定 experiment_name 来定位（见 __init__.py:392 get_recorder 的 Case 4/5：只给 id 不给实验名会报错）。
2. （流程）在 qrun config.yaml 执行过程中，YAML 的 Jinja2 模板渲染发生在解析之前还是之后？举一个会因此踩坑的例子。
参考答案
渲染发生在解析之前（run.py:106 先 render_template，再 yaml.load）。踩坑例子：YAML 写了 market: {{ MARKET }} 但没 export MARKET，os.getenv(var, "") 会把它渲染成空串，解析后 market 变成空/None，导致后续按空市场取不到股票。
3. （改错）下面代码想加载历史预测，为什么会报错？怎么改？ 
recorder = R.get_recorder(recorder_id="abc123")
pred = recorder.load_object("pred.pkl")
参考答案
缺少 experiment_name（或 experiment_id）来定位实验，在无 active recorder 情况下会报错。改为 R.get_recorder(recorder_id="abc123", experiment_name="lgb_alpha158")。另外若 pred 是由 SigAnaRecord 等带 artifact_path 的模板存的子目录文件，加载时要带前缀，如 sig_analysis/ic.pkl；但 pred.pkl 由 SignalRecord 存在根 artifact 下，直接写 pred.pkl 即可。
4. （概念）record 段里为什么 SigAnaRecord 和 PortAnaRecord 必须排在 SignalRecord 之后？它们靠什么机制避免"依赖缺失就崩溃"？
参考答案
因为二者的 depend_cls = SignalRecord，它们要读 SignalRecord 产出的 pred.pkl / label.pkl。record 是有序列表，先跑 SignalRecord 生成依赖。二者继承 ACRecordTemp（record_temp.py:212），generate() 会先 check() 依赖是否存在，缺失则告警并跳过（logger.warning），而非抛异常崩溃。

6. （思考）Qlib 为什么不让用户直接调 MLflow，而要包一层 R / Recorder？至少说出两点好处与一个代价。
参考答案
好处（见 __init__.py:3-15）：① 用 recorder 对象及其直观方法代替到处传 run_id；② 提供 save_objects/load_object 直接存取 Python 对象，无需手动 log_artifact/download_artifact；③ 启动时自动记录代码 diff，便于复现；④ 可支持多种后端。代价：设计更重，必须先创建/激活 experiment 才能拿到 recorder（MLflow 里 experiment 更像标签，很多接口直接用 run_id）。
