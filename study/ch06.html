CHAPTER 06

策略与回测 Strategy & Backtest

模型算出了每只股票的「分数」，接下来谁来决定买卖？谁来撮合成交、扣手续费、算涨跌停？本章带你走完「预测信号 → 交易决策 → 回测执行 → 组合状态」这条完整的链路。

一、从「分数」到「买卖」：策略在做什么

想象你请了一位分析师（模型），他每天早上给全市场几千只股票各打一个分——分数越高越看好。但分析师只负责「看好」，不负责「下单」。真正要回答「今天买哪几只、卖哪几只、各买多少手」的，是策略（Strategy）。

在 Qlib 里，策略的职责被抽象成一个方法：拿到当期信号，产出一组订单。所有策略都继承自 qlib/strategy/base.py:23 的 BaseStrategy，核心是这个抽象方法：

class BaseStrategy:
    def generate_trade_decision(self, execute_result: list = None):
        # 在每个交易 bar 生成交易决策（订单列表）
        # execute_result 是上一步的成交结果，首次调用可能为 None
        raise NotImplementedError

策略通过几个 property 拿到运行时所需的一切：self.trade_calendar（交易日历，当前走到第几步）、self.trade_position（当前持仓）、self.trade_exchange（交易所，用来判断可否成交、撮合订单）。这些都由回测框架在初始化时通过 common_infra / level_infra 注入。

提示策略本身不直接改持仓，它只「产出决策」。真正扣钱、加仓、算成本的是 Executor 和 Exchange。策略与执行分离，是 Qlib 回测能支持多层嵌套、多频率的关键设计。

二、signal 到底是什么：模型元组 or 现成分数

信号策略都继承自 BaseSignalStrategy（qlib/contrib/strategy/signal_strategy.py:25），构造时接受一个 signal 参数。这个参数非常灵活，内部统一交给 create_signal_from（qlib/backtest/signal.py:88）转换成 Signal 对象：

def create_signal_from(obj):
    if isinstance(obj, Signal):        # 已经是 Signal，直接用
        return obj
    elif isinstance(obj, (tuple, list)):  # (model, dataset) 元组 → 现场预测
        return ModelSignal(*obj)
    elif isinstance(obj, (dict, str)):    # 配置字典/字符串 → init_instance_by_config
        return init_instance_by_config(obj)
    elif isinstance(obj, (pd.DataFrame, pd.Series)):  # 预先算好的分数
        return SignalWCache(signal=obj)

换句话说，你有两条常见路子：

  • 传 (model, dataset) 元组：会走 ModelSignal，在初始化时立刻调用 model.predict(dataset) 把分数算出来缓存起来。examples/workflow_by_code.py:41 用的就是 "signal": (model, dataset)。
  • 传一个已经算好的 pd.Series/pd.DataFrame：走 SignalWCache，直接把分数缓存。适合你已经离线预测好、或想复用别人的预测结果。

信号的标准格式是一个 MultiIndex（instrument, datetime）的 Series，值就是分数。策略每一步会调用 self.signal.get_signal(start_time, end_time) 取出「这一期」的截面分数。

小贴士老代码里 TopkDropoutStrategy(model=..., dataset=...) 的写法已被标记为 DeprecationWarning（见 signal_strategy.py:60），内部会自动转成 signal = (model, dataset)。新项目请直接用 signal。

三、TopkDropoutStrategy：最经典的调仓逻辑

Qlib 里最常用的选股策略是 TopkDropoutStrategy。它的思路可以用一句话概括：始终持有分数最高的 topk 只股票，但每期最多只换掉 n_drop 只，避免频繁全仓换手。

两个核心参数（signal_strategy.py:81）：

  • topk：组合里始终持有的股票数量。例如 topk=50 就是满仓 50 只。
  • n_drop：每个调仓日最多换出（卖出）几只。例如 n_drop=5 表示每期卖掉当前持仓里分数最低的 5 只，再买入 5 只新的高分股票。

其它常用参数：method_sell="bottom"（卖分数最低的）、method_buy="top"（买分数最高的）、hold_thresh=1（最短持有天数）、only_tradable（是否跳过停牌不可交易的股票）、forbid_all_trade_at_limit=True（涨跌停时是否禁止一切交易）。

调仓的核心在 generate_trade_decision（signal_strategy.py:138）。我们拆解它的骨架看看它每一步在干什么：

def generate_trade_decision(self, execute_result=None):
    # 1. 定位当前是第几个交易步，取出这一期的信号
    trade_step = self.trade_calendar.get_trade_step()
    trade_start_time, trade_end_time = self.trade_calendar.get_step_time(trade_step)
    pred_start_time, pred_end_time = self.trade_calendar.get_step_time(trade_step, shift=1)
    pred_score = self.signal.get_signal(start_time=pred_start_time, end_time=pred_end_time)
    if pred_score is None:
        return TradeDecisionWO([], self)   # 当期没信号，不交易

    current_temp = copy.deepcopy(self.trade_position)   # 拷贝当前持仓
    current_stock_list = current_temp.get_stock_list()

    # 2. 把当前持仓按分数从高到低排序（last）
    last = pred_score.reindex(current_stock_list).sort_values(ascending=False).index

    # 3. 从「未持有」的股票里，选分数最高的若干只作为候选买入（today）
    today = get_first_n(
        pred_score[~pred_score.index.isin(last)].sort_values(ascending=False).index,
        self.n_drop + self.topk - len(last),
    )

    # 4. 把 last + today 合并再排序，从底部选 n_drop 只真正卖出（sell）
    comb = pred_score.reindex(last.union(pd.Index(today))).sort_values(ascending=False).index
    sell = last[last.isin(get_last_n(comb, self.n_drop))]

    # 5. 真正买入的数量 = 卖出腾出的仓位 + 还没满的仓位
    buy = today[: len(sell) + self.topk - len(last)]
    # ...对 sell / buy 逐只生成 Order，check_order 后 deal_order 撮合...
    return TradeDecisionWO(sell_order_list + buy_order_list, self)

关键巧思在第 4 步：先把「打算卖的」和「打算买的」放一起排序，只从整体底部挑 n_drop 只卖。这样能避免「卖掉高分股票、买入低分股票」的愚蠢换手——如果新候选还不如手里的，就干脆不换。

陷阱pred_score 若是多列 DataFrame，策略只会取第一列（signal_strategy.py:146）：pred_score = pred_score.iloc[:, 0]。多因子信号需要自己先合成成单列分数。

四、Exchange：成本、限价与成交价

光有订单还不够，还得有个「交易所」把订单变成真实成交，并且真实地扣手续费、挡住涨跌停。这就是 Exchange（qlib/backtest/exchange.py:28）。它的构造参数几乎就是回测里最影响收益的一组假设：

  • open_cost（默认 0.0015）：买入（开仓）成本比例，按成交额比例收取。
  • close_cost（默认 0.0025）：卖出（平仓）成本比例——A 股卖出多一道印花税，所以通常比 open_cost 高。
  • min_cost（默认 5.0）：最低佣金，绝对金额。哪怕成交额很小，也至少收 5 元（exchange.py:948：trade_cost = max(trade_val * cost_ratio, self.min_cost)）。
  • limit_threshold：涨跌停阈值。如 0.095 表示涨跌幅超过 ±9.5% 视为触及涨跌停，此时按规则禁止交易（check_stock_limit，exchange.py:338）。
  • deal_price：成交价字段。可以是单个字符串如 "close"，也可以是 (<买价>, <卖价>) 元组。字符串会自动补上 $ 前缀成为表达式 $close。

examples/workflow_by_code.py:51 里的 exchange_kwargs 是最标准的示例：

"exchange_kwargs": {
    "freq": "day",
    "limit_threshold": 0.095,   # ±9.5% 视为涨跌停
    "deal_price": "close",     # 以收盘价成交
    "open_cost": 0.0005,      # 买入万5
    "close_cost": 0.0015,     # 卖出千1.5
    "min_cost": 5,             # 最低佣金5元
}

策略调仓时会先调 trade_exchange.is_stock_tradable(...)（exchange.py:404）判断这只票当日是否停牌/触及涨跌停，不可交易就跳过；再用 check_order 校验、deal_order（exchange.py:421）真正撮合并返回成交额、成本、成交价。

务必注意成本参数看着都是小数，但复利放大后对长期收益影响极大。回测里把 close_cost 从 0.0015 调到 0.003，年化收益可能腰斩。做实盘估算时，宁可把成本设得略高，也别过度乐观。

五、Executor / Account / Position：谁来推进时间

如果说 Strategy 是「决策者」、Exchange 是「撮合台」，那 Executor（执行器） 就是「时间的推进器」。它按交易日历一步步走，每一步：向策略要决策 → 交给 Exchange 撮合 → 用成交结果更新 Account/Position → 进入下一步。

Qlib 提供两种执行器（qlib/backtest/executor.py）：

  • SimulatorExecutor（executor.py:513）：最常用的模拟执行器，逐 step 撮合。核心参数 time_per_step="day" 指定每步的频率，generate_portfolio_metrics=True 让它顺便产出净值曲线等组合指标。
  • NestedExecutor（executor.py:310）：嵌套执行器，支持「多层/多频率」。典型用法是外层日线策略决定「今天买 100 万某股票」，内层再用一个高频（如分钟级）策略/执行器把这 100 万拆单执行，模拟真实的日内成交与冲击成本。
   NestedExecutor (day)
   ┌───────────────────────────────────────────────┐
   │  日线策略: 今天买 A 100万 / 卖 B 50万           │
   │        │  把「日订单」作为 outer_trade_decision │
   │        ▼                                        │
   │   SimulatorExecutor (1min)                      │
   │   ┌───────────────────────────────────────┐    │
   │   │ 分钟策略: 把日订单拆成 240 根 bar 执行 │    │
   │   └───────────────────────────────────────┘    │
   └───────────────────────────────────────────────┘

撮合完成后，组合状态由两个对象维护：

  • Account（qlib/backtest/account.py）：账户总览，持有现金、当前 Position、并记录每步的组合指标（收益、净值）。回测入口通过 create_account_instance 用 account（一个数字或 {"cash": ..., "SH600000": {...}} 字典）初始化它。
  • Position（qlib/backtest/position.py）：持仓明细，记录每只股票的数量、成本、持有天数等。策略里的 current_temp.get_stock_list()、get_cash()、get_stock_amount(code) 都是它提供的。

而每一条「订单」用 Order（qlib/backtest/decision.py:37）表示，关键字段是 stock_id、amount、direction（Order.SELL=0 / Order.BUY=1）、start_time/end_time。策略产出的一批订单被打包成 TradeDecisionWO（Trade Decision With Order，decision.py:547）返回给执行器。

六、把三段拼起来：port_analysis_config 与 backtest 入口

整个回测的配置被组织成一个字典 port_analysis_config，它正好对应上面三大主角：executor（怎么推进）、strategy（怎么决策）、backtest（时间/账户/交易所假设）。这段配置来自 examples/workflow_by_code.py:28：

port_analysis_config = {
    "executor": {                                  # ① 执行器
        "class": "SimulatorExecutor",
        "module_path": "qlib.backtest.executor",
        "kwargs": {
            "time_per_step": "day",               # 每步一个交易日
            "generate_portfolio_metrics": True,   # 产出净值等指标
        },
    },
    "strategy": {                                  # ② 策略
        "class": "TopkDropoutStrategy",
        "module_path": "qlib.contrib.strategy.signal_strategy",
        "kwargs": {
            "signal": (model, dataset),         # (模型, 数据集) 元组
            "topk": 50,                          # 满仓持有 50 只
            "n_drop": 5,                          # 每期最多换 5 只
        },
    },
    "backtest": {                                  # ③ 回测时间/账户/交易所
        "start_time": "2017-01-01",
        "end_time": "2020-08-01",
        "account": 100000000,               # 初始资金 1 亿
        "benchmark": CSI300_BENCH,           # 基准指数（用于计算超额收益）
        "exchange_kwargs": {                  # 交给 Exchange 的参数
            "freq": "day",
            "limit_threshold": 0.095,
            "deal_price": "close",
            "open_cost": 0.0005,
            "close_cost": 0.0015,
            "min_cost": 5,
        },
    },
}

底层的编程式入口是 qlib.backtest.backtest（qlib/backtest/__init__.py:217）。它做的事是：先用 get_strategy_executor 把配置实例化成 strategy 和 executor（并注入 common_infra），再调 backtest_loop 跑完整个日历：

from qlib.backtest import backtest, executor

portfolio_metric_dict, indicator_dict = backtest(
    start_time="2017-01-01",
    end_time="2020-08-01",
    strategy=strategy_config,      # 即 port_analysis_config["strategy"]
    executor=executor_config,      # 即 port_analysis_config["executor"]
    benchmark="SH000300",
    account=100000000,
    exchange_kwargs=exchange_kwargs,
)
小贴士日常你几乎不会手写 backtest() 调用。更常见的是把 port_analysis_config 交给 PortAnaRecord（workflow_by_code.py:84），它会在实验里自动跑回测并把绩效指标记录下来。工作流与实验管理正是下一章的主题。

📝 课后练习题

1.（概念）TopkDropoutStrategy 里 topk=30、n_drop=3 分别代表什么？为什么要有 n_drop 这个「换出上限」？
参考答案
topk=30 表示组合始终持有分数最高的 30 只股票；n_drop=3 表示每个调仓日最多卖出（换出）3 只、再买入相应数量的新高分股票。设置 n_drop 是为了控制换手率：如果每期都全仓按最新分数重排，会产生巨大的交易成本，且分数的短期波动会导致频繁无意义的买卖。限制每期只动 3 只，能在「跟随信号」和「控制成本」之间取得平衡。
2.（代码理解）下面这行来自 create_signal_from，请说明什么样的 signal 参数会走到 ModelSignal，它在初始化时会立即做什么？ 
elif isinstance(obj, (tuple, list)):
    return ModelSignal(*obj)
参考答案
当 signal 是一个元组或列表（典型是 (model, dataset)）时会走 ModelSignal。它在 __init__ 里立即调用 model.predict(dataset) 算出全部预测分数，若返回的是 DataFrame 就取第一列（iloc[:, 0]），然后把分数缓存起来（继承自 SignalWCache）。因此后续每步取信号只是查缓存，不会重复预测。
3.（改错）某同学想让回测更贴近 A 股实际，写了如下 exchange_kwargs，但发现涨跌停完全没被拦截。问题出在哪？ 
"exchange_kwargs": {
    "deal_price": "close",
    "limit_thresh": 0.1,
}
参考答案
参数名拼错了。正确的键是 limit_threshold（见 qlib/backtest/__init__.py:43 的 get_exchange 签名），而不是 limit_thresh。写错的键会被 **kwargs 悄悄吞掉而不报错，导致 limit_threshold 用默认值（来自全局配置 C.limit_threshold，可能为 None，即不设涨跌停）。改成 "limit_threshold": 0.1 即可。这类「拼错参数名被静默忽略」是回测配置里非常常见的坑。
4.（概念）Strategy、Exchange、Executor 三者的职责边界是什么？为什么策略里明明有 current_temp = copy.deepcopy(self.trade_position)，却不能直接在上面改仓？
参考答案
Strategy 负责「产出决策（订单列表）」，Exchange 负责「撮合成交、扣成本、判断可交易性」，Executor 负责「按日历推进时间、驱动前两者、更新 Account/Position」。策略拿到的是持仓的深拷贝，只是用来在生成订单时做「假设性推演」（比如判断卖出后腾出多少现金），真正的持仓修改由 Executor 在撮合成交后统一完成。这种「决策与执行分离」的设计，才使得 NestedExecutor 的多层嵌套、多频率执行成为可能。
5.（思考）你有一份离线算好的预测分数 pred.pkl（MultiIndex 为 instrument/datetime 的 Series），不想再传 model。该怎样把它接入策略？内部会用哪个 Signal 类？
参考答案
直接把这个 pd.Series 作为 signal 传给策略即可，例如 TopkDropoutStrategy(signal=pred, topk=50, n_drop=5)。create_signal_from 检测到它是 pd.Series/pd.DataFrame，会用 SignalWCache 包装并缓存。这样回测就完全脱离了模型，适合复用他人预测、或把「训练/预测」与「回测」解耦成两个独立步骤。
6.（思考）什么场景下需要用 NestedExecutor 而不是普通的 SimulatorExecutor？它能帮你更真实地建模哪种成本？
参考答案
当你的下单量足够大、日内成交价格/冲击成本不可忽略时，就适合用 NestedExecutor。外层用日线策略决定「今天买卖什么、买多少」，内层再嵌套一个高频（如分钟级）执行器把日订单拆单执行。这样能建模市场冲击成本（market impact）与执行滑点——大单不可能全部按某一个价格瞬间成交。对小资金、低频策略而言普通 SimulatorExecutor 就够了；只有对交易执行本身敏感（如高频、大资金）时，多层嵌套才有价值。

📎 附录：Qlib 内置策略全览

Qlib 的策略分布在几个模块里，按用途可分成三大类。理解分类比记住类名更重要——它决定了你在什么场景该用哪个。所有类均可对照源码阅读：qlib/strategy/base.py、qlib/contrib/strategy/*.py、qlib/rl/*、qlib/workflow/online/strategy.py。

策略三大类（按用途）
────────────────────────────────────────────────────────
① 组合策略 (Portfolio)   —— 选股 + 决定买卖哪些、各多少
   信号驱动：TopkDropout / WeightStrategyBase / EnhancedIndexing / SoftTopk
   输入：模型预测分数 signal；输出：调仓订单

② 执行策略 (Execution)   —— 已知"要交易 X 股"，决定"这一天/这段时间怎么拆单成交"
   规则驱动：TWAP / SBB / AC / Random / File / SingleOrder
   输入：上层传下来的 outer_trade_decision；输出：拆分后的子订单
   常配合 NestedExecutor 用作"内层"

③ 强化学习策略 (RL)      —— 用训练好的 policy 决定动作（多为执行层）
   RLStrategy / RLIntStrategy / SAOE 系列
────────────────────────────────────────────────────────
一句话抓住区别 组合策略回答「买什么」（which stocks），执行策略回答「怎么买这批既定的量」（how to fill）。日频回测通常只用组合策略；「日频选股 + 日内执行」的嵌套回测里，外层放组合策略、内层放执行策略。所有策略都继承 BaseStrategy（base.py:23）并实现 generate_trade_decision()。

一、组合策略（信号驱动，qlib/contrib/strategy/signal_strategy.py）

策略主要逻辑关键参数应用场景
TopkDropoutStrategy 按模型分数排序，持有分数最高的 topk 只；每期把持仓中最差的 n_drop 只换成榜单上更好的 n_drop 只（"保持前 K、滚动换出 N"）。等权持仓。 topk 持仓数；n_drop 每期换手数；method_sell/buy（bottom/top/random）；hold_thresh 最短持有天数；only_tradable 是否跳过停牌；forbid_all_trade_at_limit 涨跌停是否禁交易。 最常用的基准组合策略。几乎所有 benchmark（LightGBM/GRU…）都用它。适合"选股打分 → 等权持有换手"的经典多头策略。n_drop 小则低换手低成本、反应慢；大则相反。
WeightStrategyBase 抽象基类：不直接选股，而是要求子类实现 generate_target_weight_position(score, current, ...) 给出目标权重，再由 OrderGenerator 把"当前持仓→目标权重"转成订单。 order_generator_cls_or_obj：订单生成器（默认 OrderGenWOInteract，不考虑当日成交反馈）。 当你想做非等权、按权重配置的组合时的父类。自己继承它实现权重逻辑即可，无需关心下单细节。
EnhancedIndexingStrategy 增强指数化：结合主动与被动管理，在跑赢基准指数的同时用风险模型控制跟踪误差/风险暴露。内部用二次规划优化器（optimizer/enhanced_indexing.py）在因子暴露、行业、个股偏离等约束下求最优权重。 riskmodel_root 风险模型数据路径（需 factor_exp/factor_cov/specific_risk 等）；market 基准（默认 csi500）；turn_limit 换手限制；optimizer_kwargs。 机构化的指数增强产品：既要超额收益又要严控相对基准的偏离。需要额外准备风险模型数据（可用 qlib.model.riskmodel 生成）。
SoftTopkStrategy
(cost_control.py)		 继承 WeightStrategyBase 的带成本/换手约束的 topk 权重策略：目标是每只 risk_degree/topk 的等权，但用 trade_impact_limit 限制单只每期的权重变动幅度，卖出确定性、买入按缺口比例分配预算，从而平滑换手、控制冲击成本。 topk 持仓数；risk_degree 目标投资比例（默认 0.95）；trade_impact_limit/max_sold_weight 单只每期最大权重变动。 关注交易成本与冲击的场景：不想像 TopkDropout 那样一次性大幅换仓，而是渐进调仓。

二、执行策略（规则驱动，qlib/contrib/strategy/rule_strategy.py）

这类策略不选股——它们接收上层传来的 outer_trade_decision（"今天要买/卖这些股票各多少"），负责把这批既定交易量在更细的时间粒度上拆分执行，目标是降低成交冲击、贴近某个基准价。常作为 NestedExecutor 的内层。

策略主要逻辑应用场景
TWAPStrategy 时间加权平均价（Time-Weighted Average Price）：把一笔订单的总量平均摊到剩余的每个时间步匀速成交（向上取整处理零头）。最简单、最经典的执行基准。 执行层基准算法：评估其它执行策略好坏的对照；也用于"不想暴露交易意图、匀速铺量"的场景。
SBBStrategyBase 抽象基类——(S)elect the (B)etter one among every two adjacent (B)ars：在相邻两根 K 线中择优成交。奇数步先预测价格趋势（TREND_SHORT/MID/LONG），据此决定在两根 bar 中的哪一根多买/多卖。子类需实现 _pred_price_trend。 比 TWAP 更聪明的执行框架：想利用短期趋势择时下单，但只在"两两相邻 bar"的局部做选择。
SBBStrategyEMA SBB 的具体实现，用 EMA 双均线信号 EMA($close,10)-EMA($close,20) 判断价格趋势方向，再据 SBB 逻辑择优成交。 开箱即用的"趋势择时执行"：认为短期均线金叉/死叉能指示当根 bar 该多买还是多卖。
ACStrategy Almgren-Chriss 最优执行模型：在市场冲击成本与价格波动风险之间做权衡，用波动率（对数收益的滚动标准差）和风险厌恶参数求解最优成交节奏。lamb 风险厌恶、eta 冲击系数、window_size 波动率窗口。 学术界经典的最优交易执行：大单拆分时，既怕冲击成本（拆太快）又怕价格漂移风险（拆太慢），用数学模型求平衡。
RandomOrderStrategy 按 sample_ratio 从股票池随机抽样，以历史均量的 volume_ratio 比例随机下单。 测试/压力/数据生成：构造随机交易流来测试执行环境、生成 RL 训练数据，而非真实盈利策略。
FileOrderStrategy 从 CSV / DataFrame 读取预先写好的订单（datetime, instrument, amount, direction）直接执行。 回放外部订单：把别处（其它系统、人工、离线优化）产生的订单清单导入 qlib 回测；或复现固定交易序列。
SingleOrderStrategy
(rl/strategy/single_order.py)		 只生成恰好一笔订单的极简策略。 单订单执行研究（尤其 RL 订单执行）的最小载体：把"执行这一笔单"作为一个可被优化的环境。

三、强化学习策略（qlib/strategy/base.py + qlib/rl/）

RL 策略把"决策"交给训练好的 policy（策略网络），主要用于订单执行层（Single-Asset Order Execution, SAOE），需安装 rl 依赖。

策略主要逻辑应用场景
RLStrategy RL 策略基类：持有一个 policy，其余交由子类。 所有 RL 策略的父类。
RLIntStrategy (RL)+(Int)erpreter：加入 state_interpreter（把回测执行结果翻译成 RL 状态）与 action_interpreter（把 policy 动作翻译成订单）。决策流程：解释状态 → policy 出动作 → 解释成订单。 把训练好的 RL agent接入 qlib 回测/执行的通用桥梁。
SAOEStrategy / SAOEIntStrategy / ProxySAOEStrategy
(rl/order_execution/strategy.py)		 面向单资产订单执行的 SAOE 系列：在给定"要在某时间窗内成交 X 股"的目标下，用 RL 决定每一步的成交量以最小化成本。ProxySAOEStrategy 提供进程/环境代理。 智能订单执行研究与落地（见 examples/rl_order_execution/）：让 RL 学出比 TWAP/AC 更优的拆单节奏。

四、在线服务策略（qlib/workflow/online/strategy.py）

注意：这里的"策略"是另一个维度——不是交易决策，而是模型/任务的在线更新策略，配合 Online Manager 使用。

策略主要逻辑应用场景
OnlineStrategy 抽象基类：定义"任务如何生成、模型如何更新、信号如何准备"的在线运维流程。 在线服务的编排接口。
RollingStrategy 滚动更新：随时间推移周期性用最新数据重新训练/滚动模型任务，应对市场分布漂移。 模型上线后的持续迭代：定期滚动重训，保持模型对最新行情的适应力（对比"训练一次用到底"）。
怎么选
  • 做选股回测（大多数情况）→ TopkDropoutStrategy；想要按权重/控成本 → WeightStrategyBase 子类 / SoftTopkStrategy；要指数增强 → EnhancedIndexingStrategy。
  • 做成交执行研究（拆单） → 先用 TWAPStrategy 打基准，再比 SBBStrategyEMA / ACStrategy / RL 的 SAOE 系列。
  • 要回放外部订单 → FileOrderStrategy；要随机数据 → RandomOrderStrategy。
  • 要模型上线滚动更新 → RollingStrategy（这是运维策略，不是交易策略）。
