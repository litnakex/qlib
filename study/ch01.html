CHAPTER 01

Qlib 简介与架构概览

在写第一行代码之前，先看清全局：量化研究到底在做什么，Qlib 站在什么位置，以及它那条从「数据」流到「分析」的流水线长什么样。

一、量化投资是在做什么？

先讲个生活里的场景。你去菜市场买菜，凭经验知道「早上的菜新鲜」「雨天青菜会涨价」——这是主观判断。而如果你把过去三年每天的菜价、天气、进货量全部记成表格，写一个程序自动算出「明天哪种菜最值得囤」，这就是量化。

把「菜」换成「股票」，把「囤货赚差价」换成「低买高卖赚收益」，量化投资的骨架就出来了：用数据和数学模型，代替拍脑袋，去做买卖决策。它通常包含四个动作：

  • 找信号（预测）：从海量历史行情里挖出「哪些股票未来可能涨」的规律，输出一个打分（signal / score）。
  • 做决策（策略）：根据打分决定「买哪些、买多少、什么时候卖」。
  • 模拟验证（回测）：拿历史数据当「时光机」，假装从过去某天开始按策略操作，看最后赚没赚钱。
  • 复盘分析：算出收益率、回撤、夏普比率等指标，判断策略到底好不好。
提示量化研究的核心痛点不是「写模型」，而是把数据、特征、模型、回测这一整条链路串起来还不出错。Qlib 存在的最大意义，就是把这条又长又容易踩坑的链路做成了标准化、可复用的积木。
陷阱回测里最致命的错误叫「未来函数 / 前视偏差（look-ahead bias）」——不小心用到了「当时还不知道」的信息，导致回测收益虚高、实盘一败涂地。Qlib 专门用 qlib/data/pit.py（Point-In-Time 数据）来防这类问题，后面章节会展开。

二、Qlib 是什么？它的定位

Qlib 是微软（Microsoft）开源的一套面向 AI 的量化投资研究平台。虽然 GitHub 仓库叫 qlib，但你 pip install 时装的包名是 pyqlib，代码里则一律 import qlib。

它的定位不是「一个模型」或「一个回测器」，而是覆盖了量化研究完整流水线的一整套工具箱。用官方 CLAUDE.md 里的原话（CLAUDE.md:7）：它涵盖 data ingestion（数据接入）、feature/factor computation（因子计算）、model training（模型训练）、portfolio strategy（组合策略）、backtest execution（回测执行）到 analysis（分析）。

更关键的一句设计理念也在同一处："Components are intentionally loose-coupled — each can be used stand-alone." 组件之间刻意做成松耦合，每一块都能单独拿出来用。这意味着：

  • 你可以只用它的数据层（高性能读取 A 股/美股行情），模型和策略全用自己的。
  • 也可以只用它的回测引擎，信号来自你自己的模型。
  • 当然也可以从头到尾全用 Qlib——这正是新手最省心的方式。
维度Qlib 的做法
开源方Microsoft，MIT 许可证
安装包名pyqlib（导入时是 qlib）
覆盖范围数据 → 数据集 → 模型 → 策略 → 回测 → 记录 → 分析（全流程）
耦合方式组件松耦合，可单独使用
面向对象量化研究员、想把 AI/ML 用到投资里的开发者
小贴士Qlib 自带两个 C 语言写的 Cython 扩展（qlib/data/_libs/rolling.pyx、expanding.pyx），用来加速滚动窗口计算。所以光 pip install -e . 是不够的，必须先编译它们（make install 或 make prerequisite）。这点第 2 章会详细讲。

三、核心设计哲学：一切皆可「配置字典」

如果说 Qlib 有一个「一以贯之」的设计精髓，那就是这句话：几乎所有东西，都能用一个嵌套字典来描述并实例化。这个字典长这样：

{
    "class": "LGBModel",                                # 要实例化的类名
    "module_path": "qlib.contrib.model.gbdt",          # 这个类在哪个模块里
    "kwargs": {"loss": "mse", "learning_rate": 0.05}  # 传给构造函数的参数
}

这个字典交给 qlib.utils.init_instance_by_config 就能变成一个真正的 Python 对象。我们直接看它的签名（qlib/utils/mod.py:122）：

def init_instance_by_config(
    config: InstConf,
    default_module=None,
    accept_types: Union[type, Tuple[type]] = (),
    try_kwargs: Dict = {},
    **kwargs,
) -> Any:
    # 如果 config 已经是目标类型的实例，直接返回（accept_types）
    # 如果 config 是 'file://xxx.pkl' 字符串，就反序列化加载
    # 否则：根据 class + module_path 找到类，用 kwargs 实例化
    klass, cls_kwargs = get_callable_kwargs(config, default_module=default_module)
    return klass(**cls_kwargs, **try_kwargs, **kwargs)

为什么这个小小的机制如此重要？因为它是把所有组件粘在一起的「胶水」。CLAUDE.md:62 里点明了这一点：只要你的新模型/策略/处理器「可以通过 module_path + class 被导入」，YAML 配置和 Python 代码就能用同一套机制加载它——不需要改动任何框架代码，组件就能随意替换。这就是「松耦合」在工程上的落地方式。

两种等价的使用接口

正因为有了「配置字典」这层抽象，Qlib 提供了两种做同一件事的接口，你可以按喜好选择：

配置驱动（YAML）代码驱动（Python）
入口qrun config.yaml手写脚本，见 examples/workflow_by_code.py
本质把 YAML 解析成配置字典直接在代码里写配置字典
适合跑基准、批量实验、少改代码调试、理解流程、深度定制
底层最终都走 init_instance_by_config，殊途同归

下面是 examples/workflow_by_code.py 里最能体现「配置即对象」的两行代码（第 25~26 行）——一个字典进去，一个模型/数据集出来：

from qlib.utils import init_instance_by_config
from qlib.tests.config import CSI300_GBDT_TASK

model   = init_instance_by_config(CSI300_GBDT_TASK["model"])    # 配置字典 → 模型对象
dataset = init_instance_by_config(CSI300_GBDT_TASK["dataset"])  # 配置字典 → 数据集对象
记住这句话当你以后在 YAML 里看到成堆的 class / module_path / kwargs，不要慌——它们只是 init_instance_by_config 的「原料」。理解了这一点，整个 Qlib 的配置体系就通了一大半。

四、整体架构与数据流

CLAUDE.md:60 用一句话概括了 Qlib 的工作流方向：data → dataset/handler → model → signal → strategy → executor (backtest) → recorder → analysis。这条链路可以画成下面这张图——数据从左边流入，决策与分析从右边流出：

   ┌──────────┐   原始行情/因子      ┌─────────────────────┐
   │  数据层   │  ───────────────▶ │  Handler / Dataset  │
   │  (D 提供者)│   .bin 高性能读取   │  加载特征 + 处理器    │
   │ qlib/data │                   │  切分 train/valid/test│
   └──────────┘                   └──────────┬──────────┘
                                             │ 特征矩阵 X, 标签 y
                                             ▼
                                     ┌───────────────┐
                                     │     模型      │  fit(dataset)
                                     │  qlib/model  │  predict(dataset)
                                     └──────┬───────┘
                                            │ signal（每只股票的打分）
                                            ▼
   ┌──────────┐    交易决策         ┌───────────────┐
   │  执行器   │ ◀──────────────── │     策略      │  例：TopkDropout
   │ 回测/撮合 │   下单/成交/持仓     │ qlib/strategy │  打分 → 买卖清单
   │qlib/backtest│                 └───────────────┘
   └────┬─────┘
        │ 组合净值/交易明细
        ▼
   ┌──────────────┐   收益率/回撤/夏普   ┌──────────────┐
   │   Recorder    │ ────────────────▶ │    分析报告   │
   │ 记录实验(MLflow)│  SignalRecord/    │  信号/组合分析 │
   │ qlib/workflow │  SigAnaRecord等    └──────────────┘
   └──────────────┘

把它对应到人话就是：先把数据整理成「特征表」→ 喂给模型学出「打分」→ 策略根据打分决定买卖 → 执行器在模拟市场里成交 → 记录器把整个实验存下来 → 最后算指标做分析。本教程后面 7 章，几乎就是沿着这条箭头逐段深入。

对照代码看examples/workflow_by_code.py 的第 68~85 行就是这条链路的完整代码版：model.fit(dataset) 训练 → SignalRecord 生成预测 → SigAnaRecord 做信号分析 → PortAnaRecord 做组合回测分析。全都包在 with R.start(...) 这个「实验记录上下文」里。

五、顶层包的模块地图

初学者面对一个大项目最容易迷路。这里给你一张「地图」，把 qlib/ 目录下每个顶层包的职责一次讲清（对应 CLAUDE.md 的 Architecture 一节）。后面每一章基本就是深入其中一两个包：

包 / 模块职责（一句话）本教程章节
qlib/data/数据提供层。data.py（D 提供者 API）、ops.py（因子表达式算子，底层是 Cython）、cache.py（H 内存缓存，qlib.init 会清空它）、storage/（.bin 磁盘格式）、pit.py（防前视的时点数据）。第 3 章
qlib/data/dataset/数据集与处理器。handler.py（DataHandler/DataHandlerLP 加载特征+跑处理器）、loader.py、processor.py（归一化/填充），以及把 handler 与「训练/验证/测试时间段」配对的 DatasetH。第 4 章
qlib/model/模型层。base.py 定义 Model 抽象基类（fit(dataset) / predict(dataset, segment)）；具体模型在 qlib/contrib/model/；还有 trainer.py、riskmodel/、meta/、ens/。第 5 章
qlib/strategy/策略基类 base.py；把模型信号翻译成买卖决策（如 TopkDropoutStrategy，在 qlib/contrib/strategy/）。第 6 章
qlib/backtest/回测执行引擎。exchange.py（交易成本/涨跌停建模）、executor.py（SimulatorExecutor、多层 NestedExecutor）、account.py/position.py（组合状态）、decision.py（订单）。第 6 章
qlib/workflow/实验管理。单例 R（qlib/workflow/__init__.py）封装 MLflow：R.start()、R.log_params、R.save_objects；record_temp.py 里的 SignalRecord/SigAnaRecord/PortAnaRecord 是可复用的分析步骤。第 7 章
qlib/contrib/「电池仓」——开箱即用的模型、策略、数据处理器、评估、报告、滚动与调参工具。新的用户级实现通常放这里，而不是核心包。全程
qlib/rl/强化学习框架（订单执行等）：simulator.py、interpreter.py、reward.py、order_execution/。需要额外安装 rl extra。进阶
qlib/config.py全局配置对象 C。default_conf 为 client 或 server；由 qlib.init / qlib.auto_init 完成设置。第 2 章
一个约定凡是你自己写的新模型/策略/处理器，都应该放到 qlib/contrib/ 下，并遵守 init_instance_by_config 契约（可导入的类 + kwargs）。核心包（qlib/data、qlib/model 等）尽量别动——这就是「框架」与「插件」的分界线。

六、最短的「Hello, Qlib」

不用背，先感受一下从初始化到跑通全流程的最小骨架（简化自 examples/workflow_by_code.py）：

import qlib
from qlib.constant import REG_CN
from qlib.utils import init_instance_by_config, flatten_dict
from qlib.workflow import R
from qlib.workflow.record_temp import SignalRecord, SigAnaRecord, PortAnaRecord
from qlib.tests.config import CSI300_GBDT_TASK

# 1. 初始化：告诉 Qlib 数据在哪、用哪个市场（REG_CN=中国A股）
qlib.init(provider_uri="~/.qlib/qlib_data/cn_data", region=REG_CN)

# 2. 配置字典 → 对象
model   = init_instance_by_config(CSI300_GBDT_TASK["model"])
dataset = init_instance_by_config(CSI300_GBDT_TASK["dataset"])

# 3. 在一个实验记录上下文里：训练 → 预测 → 分析
with R.start(experiment_name="workflow"):
    model.fit(dataset)                    # 训练
    recorder = R.get_recorder()
    SignalRecord(model, dataset, recorder).generate()  # 生成预测信号
    SigAnaRecord(recorder).generate()                  # 信号分析（如 IC）

看不懂每一行没关系——你只需要认出这条熟悉的链路：init → 造 model/dataset → fit → 记录 → 分析。这正是第四节那张架构图的代码化身。全书接下来要做的，就是把这几行里的每个环节掰开揉碎。

前置条件上面第 1 步的 qlib.init(...) 必须在任何数据/模型 API 之前调用，它负责设置全局配置 C、清空内存缓存 H。而数据（~/.qlib/qlib_data/cn_data）不在仓库里，需要用 scripts/get_data.py 单独下载。这两件事都是第 2 章的主题。

📝 课后练习题

1.（概念）用你自己的话说清楚 Qlib 工作流的七个环节，并按数据流方向排序。
参考答案
顺序为：data（数据）→ dataset/handler（数据集与处理器）→ model（模型）→ signal（信号/打分）→ strategy（策略）→ executor（回测执行）→ recorder（记录）→ analysis（分析）。直观理解：整理数据成特征表 → 模型学出打分 → 策略把打分变成买卖 → 执行器在模拟市场成交 → 记录器保存实验 → 最后算指标做分析。见 CLAUDE.md:60。
2.（概念）Qlib 说组件是「松耦合、可单独使用」，请举一个「只用其中一部分」的实际场景。
参考答案
例如：只用 qlib/data 数据层来高性能读取 A 股行情、计算因子，而模型和回测全用自己写的；或者反过来，信号由你自己的模型产生，只借用 qlib/backtest 的回测引擎来做撮合与成本建模。松耦合意味着每个包都能拆出来独立用。
3.（代码填空）补全下面把配置字典变成模型对象的代码。 
from qlib.utils import ________
cfg = {"class": "LGBModel",
       "module_path": "qlib.contrib.model.gbdt",
       "kwargs": {"loss": "mse"}}
model = ________(cfg)
参考答案
两处都填 init_instance_by_config：
from qlib.utils import init_instance_by_config
model = init_instance_by_config(cfg)
它会读取 module_path+class 找到类，再用 kwargs 实例化。定义见 qlib/utils/mod.py:122。
4.（改错）新同事写道：「我在 examples/ 里调 D.features(...) 读数据前，不需要调用 qlib.init()。」这句话对吗？
参考答案
不对。qlib.init(...) 必须在使用任何数据/模型 API 之前调用。它设置全局配置对象 C、注册 provider、并清空内存缓存 H（见 qlib/__init__.py 的 init 函数）。不 init 就用数据接口会报错或行为异常。
5.（辨析）qrun config.yaml 和写 workflow_by_code.py 脚本，两者是什么关系？
参考答案
二者是等价的两种接口，做同一件事。qrun 是配置驱动：把 YAML 解析成配置字典；脚本是代码驱动：直接在 Python 里写配置字典。但两条路最终都通过 init_instance_by_config 实例化组件、跑同一条流水线。配置驱动适合批量跑基准，代码驱动适合调试与定制。见 CLAUDE.md:53-56。
6.（思考）我写了一个全新的选股策略类，应该放到 qlib/ 下的哪个位置？为什么？需要满足什么条件才能被配置系统调用？
参考答案
应放到 qlib/contrib/strategy/（用户级实现放 contrib，核心包不动）。要被配置系统调用，必须遵守 init_instance_by_config 契约：类可以通过 module_path + class 被导入，且构造参数能用 kwargs 传入。满足这一点后，YAML 和代码都能无侵入地加载它——这正是松耦合可扩展的关键。
