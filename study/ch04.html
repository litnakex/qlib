CHAPTER 04

数据集与处理器 Dataset & Handler

从磁盘上的原始行情，到模型能直接吃进去的干净张量——这中间的「厨房」就是 Handler 与 Dataset。本章带你看清这道桥梁的每一块砖。

1. 为什么需要「桥梁」：从菜市场到餐盘

想象你要做一道菜。菜市场里的原料（原始行情数据 $open、$close、$volume）不能直接端上桌：蔬菜要洗、要切、要去掉烂叶，还要按菜谱把它们组合成「菜」（因子/特征）。 在 Qlib 里，把「买菜」交给上一章讲的数据层（Data Layer），而洗切摆盘这一步，就交给本章的两位主角：

  • Handler（数据处理器/数据管家）：负责「加载 → 算因子 → 清洗标准化」，产出一张干净的、带 feature / label 两组列的宽表（DataFrame）。
  • Dataset（数据集）：在 Handler 之上，负责按时间段把数据切成 train / valid / test，喂给模型。
   原始行情            算因子/清洗              按时间切分
 ┌──────────┐  load  ┌──────────────┐  fetch  ┌───────────────┐
 │ Data     │ ─────▶ │  DataHandler │ ──────▶ │   DatasetH    │
 │ Layer    │        │ (feature+    │         │ train/valid/  │ ──▶ Model
 │ ($close…)│        │  label 宽表) │         │ test 三段     │
 └──────────┘        └──────────────┘         └───────────────┘
     菜市场              后厨(洗切摆盘)             分餐盘上桌
提示 记住这条主线：DataLoader 加载 → DataHandler 处理 → DatasetH 切分。本章聚焦后两步，源码主要在 qlib/data/dataset/handler.py、qlib/data/dataset/processor.py、qlib/data/dataset/__init__.py。

返回的宽表长这样（多级列索引，外层是「组名」feature / label，内层是具体字段），见 qlib/data/dataset/handler.py:89 的文档示例：

                          feature                              label
                          $close     $volume  Ref($close,1)   LABEL0
datetime   instrument
2010-01-04 SH600000    81.807068  17145150.0   83.737389     0.0032
           SH600004    13.313329  11800983.0   13.313329     0.0042

2. Handler 的三层继承：ABC → DataHandler → DataHandlerLP

翻开 qlib/data/dataset/handler.py，你会看到三层由抽象到具体的类，像「接口 → 通用实现 → 高级实现」：

 DataHandlerABC   只定义 fetch() 接口，不假设内部结构   (handler.py:25)
        │
 DataHandler      用一个 DataFrame 实现，含 loader/     (handler.py:67)
        │         setup_data/fetch/config
        │
 DataHandlerLP    LP = Learnable Processor，          (handler.py:382)
                  维护 3 份数据 + 可学习的处理链
  • DataHandlerABC（抽象接口）：只声明一个抽象方法 fetch(selector, level, col_set, data_key)，规定「怎么取数」的契约，不管数据怎么存。
  • DataHandler（基础实现）：内部持有一个 self._data（由 DataLoader 加载得到），实现 setup_data() 加载、fetch() 取数、config() 改配置。
  • DataHandlerLP（可学习处理器版）：最常用的一层。它在原始数据之上，跑一条「处理链」，同时产出推理用和训练用两份不同的数据。

DK_R / DK_I / DK_L：一份原料，三种成品

DataHandlerLP 会把数据维护成三份（见 handler.py:417-422），用三个「data key」区分：

  • DK_R = "raw" → self._data：DataLoader 直接加载的原始数据。
  • DK_I = "infer" → self._infer：经过 infer_processors 处理、给推理/预测用的数据。
  • DK_L = "learn" → self._learn：经过 learn_processors 处理、给模型训练用的数据。
类比 同一批食材：给客人试吃的摆盘（infer，尽量还原真实场景），和厨师练习刀工时的版本（learn，可以扔掉不合格的样本、做更激进的归一化）——原料一样，但目的不同，所以处理方式不同。

3. infer_processors vs learn_processors：为什么要分家？

这是初学者最容易困惑的地方。为什么要维护两条处理链？核心原因是「推理时能拿到的信息」和「训练时能利用的信息」不一样，见 handler.py:396-402 的动机说明：

  • learn 阶段可以「作弊」利用 label：比如 DropnaLabel 会把没有标签（未来收益缺失）的样本丢掉——训练时丢掉合理，因为没标签的样本没法算 loss。
  • infer 阶段是真实预测场景，此时还不知道未来的 label，所以任何依赖 label 的处理都不能用。

因此 Qlib 用 is_for_infer() 给处理器打标签：像 DropnaLabel 就返回 False（见 processor.py:109），一旦有人把它放进 infer_processors，_run_proc_l 会直接报错（handler.py:534）。

两种拼接方式：PTYPE_I 与 PTYPE_A

还有一组 shared_processors（共享处理链），它总是先跑。之后 infer 和 learn 怎么接，由 process_type 决定（handler.py:552-609）：

PTYPE_I ("independent")  独立：
  (raw)-[shared]-(shared_df)-[infer]-(infer_df)
                        \
                         -[learn]-(learn_df)

PTYPE_A ("append")  追加（Alpha158/360 默认）：
  (raw)-[shared]-[infer]-(infer_df)-[learn]-(learn_df)
        learn 建立在 infer 的输出之上，再追加 learn 处理
陷阱 PTYPE_A 意味着 learn 数据 = infer 数据再经过 learn_processors。所以像标准化这种放在 infer 里的处理，learn 数据也会继承。理解这一点，才能看懂 Alpha158 里 infer_processors 为空、只靠 learn 阶段加 DropnaLabel + label 归一化的设计。

关键方法速查（signatures）

4. Processor 家族：洗切摆盘的具体工具

处理器都继承自 Processor（processor.py:35），有两个核心方法：

  • fit(df)：学习处理参数（如均值、标准差）。默认空实现，需要参数的处理器才重写。
  • __call__(df)：真正应用处理，返回处理后的 DataFrame（注意：可能原地修改！）。

另外两个辅助方法：is_for_infer()（能否用于推理）、readonly()（是否只读，帮 Handler 决定要不要 copy）。常用处理器一览（均来自 processor.py）：

处理器作用要 fit?行号
Fillna用固定值（默认 0）填 NaN否179
DropnaProcessor按某组列丢掉含 NaN 的行否94
DropnaLabel专丢 label 为空的样本，仅训练用否105
ProcessInf把 ±inf 替换为该列均值否161
ZScoreNorm标准化 (x-mean)/std是228
RobustZScoreNorm用中位数/MAD 的稳健标准化，可裁剪离群值是262
MinMaxNorm归一化到 [0,1]是196
CSZScoreNorm横截面（每日）ZScore否300
CSRankNorm横截面按天排名再归一化否326

核心陷阱：fit 只能在训练集上做，避免数据泄露

看 ZScoreNorm 的构造函数（processor.py:231）：它要求传入 fit_start_time 与 fit_end_time。fit() 里会先把数据裁到这段时间再计算均值方差： 

def fit(self, df=None):
    df = fetch_df_by_index(df, slice(self.fit_start_time, self.fit_end_time), level="datetime")
    cols = get_group_columns(df, self.fields_group)
    self.mean_train = np.nanmean(df[cols].values, axis=0)  # 只用训练区间算均值
    self.std_train  = np.nanstd(df[cols].values, axis=0)
数据泄露警告 如果 fit_end_time 覆盖到了测试集，测试期的均值/方差就「偷看」进了标准化参数——回测会虚高、上线暴雷。源码注释反复强调：fit_end_time 绝不能包含任何测试数据的信息（processor.py:232-233）。正确做法：在训练集上 fit，再把同一套参数应用到 valid/test。

而 CSRankNorm / CSZScoreNorm 是横截面操作——每天内部对所有股票做排名/标准化（groupby("datetime")），不依赖历史统计量，所以不需要 fit，也就没有泄露风险。

5. Alpha158 与 Alpha360：开箱即用的因子集

每次都手写几百个因子太累。Qlib 在 qlib/contrib/data/handler.py 里预置了两个继承自 DataHandlerLP 的现成 Handler：

  • Alpha158（handler.py:98）：约 158 个「加工好的」因子。特征配置 = K 线形态（kbar）+ 部分价格量（OPEN/HIGH/LOW/VWAP）+ 大量滚动统计量（rolling，如均线、波动、动量）。每支股票每天是一行、多列特征——是树模型（LightGBM 等）的常用输入。
  • Alpha360（handler.py:48）：360 个「原始」因子 = 过去 60 天的 close/open/high/low/vwap/volume（6 × 60 = 360）。它基本是原始行情的时间窗切片，更适合能自己学特征的深度模型（GRU/LSTM/Transformer）。
一句话区分 Alpha158 = 「人工特征工程」后的因子（信息密度高，喂树模型）；Alpha360 = 「原始 60 天历史」的滑窗（让神经网络自己提特征）。

看它们的默认处理链设计（handler.py:37-45），能把前面所有概念串起来：

# Alpha360 的默认 infer 处理：先处理 inf → 标准化 → 填 NaN
_DEFAULT_INFER_PROCESSORS = [
    {"class": "ProcessInf"}, {"class": "ZScoreNorm"}, {"class": "Fillna"},
]
# learn 处理：丢掉无标签样本 + 对 label 做横截面 ZScore（仅训练用）
_DEFAULT_LEARN_PROCESSORS = [
    {"class": "DropnaLabel"},
    {"class": "CSZScoreNorm", "kwargs": {"fields_group": "label"}},
]

注意 Alpha158 默认 infer_processors=[]、process_type=PTYPE_A（handler.py:105-109），把标准化交给 config 里传入的处理器。而 check_transform_proc（handler.py:12）会自动把 fit_start_time/fit_end_time 注入到需要它们的处理器中——这正是前面「只在训练集 fit」的落地机制。

6. DatasetH：把 Handler 与时间段组合起来

DatasetH（qlib/data/dataset/__init__.py:72）= Dataset with (H)andler。它只做两件事：持有一个 handler，加一个 segments 时间切分字典。 

def __init__(self, handler, segments, fetch_kwargs={}, **kwargs):
    self.handler  = init_instance_by_config(handler, accept_types=DataHandler)
    self.segments = segments.copy()   # 如 {"train": (...), "valid": (...), "test": (...)}
别搞混 segments（train/valid/test）是时间维度的切分；DK_I/DK_L是处理方式的切分。两者正交！你完全可以「取 train 时间段、但用 infer 处理链（DK_I）」。源码注释专门澄清了这点（handler.py:410-414）。

prepare(...)：取出可用于建模的数据

模型通过 dataset.prepare(...) 拿数据（__init__.py:185）。签名与真实用法：

def prepare(self, segments, col_set=CS_ALL, data_key=DK_I, **kwargs)

# 取训练集：feature+label 两组列，用 learn 处理链（含 DropnaLabel）
df_train = dataset.prepare("train", col_set=["feature", "label"], data_key=DataHandlerLP.DK_L)
# 取测试集：用 infer 处理链（不丢样本，还原真实推理）
df_test  = dataset.prepare("test",  col_set=["feature", "label"], data_key=DataHandlerLP.DK_I)
# 一次取多段
df_tr, df_va = dataset.prepare(["train", "valid"])

这段用法正是 qlib/model/base.py 的 Model.fit 文档字符串里给出的范式：训练用 DK_L、预测用 DK_I。内部 prepare 会把段名映射成时间 slice，再调用 handler.fetch(slc, col_set=..., data_key=...)（__init__.py:229-247）。X = df["feature"]、y = df["label"] 即可拆出特征与标签。

用 init_instance_by_config 构造 Alpha158 DatasetH

实战中几乎不手写 Python 构造，而是写 YAML 交给 init_instance_by_config（第 3 章讲过）。下面片段摘自 examples/benchmarks/LightGBM/workflow_config_lightgbm_Alpha158.yaml：

📝 课后练习题

1.（概念）DataHandlerLP 维护的 DK_R / DK_I / DK_L 三份数据分别是什么？谁给谁当输入？
参考答案
DK_R (raw, self._data) 是 DataLoader 加载的原始数据；DK_I (infer, self._infer) 是经 infer_processors 处理、供推理用；DK_L (learn, self._learn) 经 learn_processors 处理、供训练用。三者都先经 shared_processors。默认 PTYPE_A 下，learn 数据建立在 infer 输出之上再追加 learn 处理；PTYPE_I 下两条链各自独立从 shared 输出分叉。
2.（概念）为什么 DropnaLabel 只能放进 learn_processors，不能放 infer_processors？
参考答案
它按 label 是否为空来丢样本。推理（infer）是真实预测场景，此时未来 label 未知，依赖 label 的处理不合法且会丢掉本该预测的样本。源码里 DropnaLabel.is_for_infer() 返回 False（processor.py:109），_run_proc_l 在处理 infer 链时用 check_for_infer=True 检查，遇到它会抛 TypeError（handler.py:534）。
3.（改错）某同学把 ZScoreNorm 配成 fit_start_time=2008-01-01, fit_end_time=2020-08-01，而 test 段是 2017-2020。问题在哪？怎么改？
参考答案
数据泄露：fit_end_time 覆盖到了 test 区间，标准化用的均值/方差偷看了测试期数据，导致回测虚高、实盘失效。应把 fit_end_time 限制到 train 段末尾（如 2014-12-31），保证只在训练集上 fit，再把同一套参数应用到 valid/test。见 processor.py:232-233 的警告。
4.（概念）Alpha158 与 Alpha360 的本质区别是什么？各自更适合哪类模型？
参考答案
Alpha158 是人工特征工程后的约 158 个因子（kbar 形态 + 价格 + 大量 rolling 统计量），信息密度高，适合 LightGBM 等树模型；Alpha360 是过去 60 天 × 6 个价量字段 = 360 个「原始滑窗」特征，几乎是原始行情，适合能自动提特征的深度模型（GRU/LSTM/Transformer）。见 contrib/data/handler.py。
5.（填空）补全取训练/测试数据的调用，使其符合「训练用 learn 链、测试用 infer 链」的规范：
tr = ds.prepare("train", col_set=["feature","label"], data_key=____)
te = ds.prepare("test", col_set=["feature","label"], data_key=____)
参考答案
训练用 DataHandlerLP.DK_L（即 "learn"），测试用 DataHandlerLP.DK_I（即 "infer"）。prepare 默认 data_key=DK_I，所以测试那行不写也行；但训练必须显式指定 DK_L，否则拿不到 DropnaLabel/label 归一化后的训练数据。
6.（思考）segments 的 "train/valid/test" 和 data_key 的 DK_I/DK_L 是同一个维度的切分吗？举例说明二者可以自由组合。
参考答案
不是。segments 是时间维度切分（哪段日期），data_key 是处理方式切分（跑哪条处理链）。二者正交，可任意组合：例如 prepare("train", data_key=DK_I) 表示「取训练时间段、但用推理处理链」，常用于观察训练期在推理设定下的表现。源码 handler.py:410-414 专门澄清这两个概念不同。

📎 附录：Alpha158 与 Alpha360 因子全表

下面逐一列出两套因子集里每个因子的名称、qlib 表达式、含义与计算方法。表达式全部摘自源码 qlib/contrib/data/loader.py（Alpha158DL / Alpha360DL），可对照阅读。

先记住几条通用约定
  • $open/$high/$low/$close/$volume/$vwap 是原始行情字段（开/高/低/收/成交量/成交均价）。
  • Ref($x, d)：d 天前的 $x（d>0 回看历史；d<0 看未来，仅用于 label）。
  • 大量因子都除以 $close（或 $volume）做归一化——目的是去掉量纲，让不同价位的股票可比。所以「均线 MA5」在 qlib 里是 Mean($close,5)/$close 而非绝对均价。
  • +1e-12 是防止分母为零的保护项，无实际含义。
  • 滚动窗口默认取 d ∈ {5, 10, 20, 30, 60}（约一周~一季度）。

一、Alpha158（约 158 个「加工好」的因子）

Alpha158 默认配置（handler.py:140-149）= 9 个 K 线形态 + 4 个当日价格 + 29 类 × 5 窗口 = 145 个滚动统计量，合计 158。它信息密度高，是 LightGBM 等树模型的常用输入。

1. K 线形态因子（kbar，9 个）——刻画单根 K 线的形状

名称表达式含义 / 计算
KMID($close-$open)/$open实体涨跌幅：收盘相对开盘的涨跌，衡量当日多空强弱。
KLEN($high-$low)/$openK 线全长（振幅）：最高最低之差占开盘价的比例，衡量当日波动幅度。
KMID2($close-$open)/($high-$low+1e-12)实体占振幅的比例：实体在整根 K 线里占多大，接近 ±1 表示单边、接近 0 表示上下影线长。
KUP($high-Greater($open,$close))/$open上影线长度 / 开盘价：Greater(a,b) 取较大者，即实体顶端到最高价的距离，反映上方抛压。
KUP2($high-Greater($open,$close))/($high-$low+1e-12)上影线占振幅的比例。
KLOW(Less($open,$close)-$low)/$open下影线长度 / 开盘价：Less(a,b) 取较小者，即实体底端到最低价的距离，反映下方承接。
KLOW2(Less($open,$close)-$low)/($high-$low+1e-12)下影线占振幅的比例。
KSFT(2*$close-$high-$low)/$open收盘在当日区间中的位置偏移 / 开盘价：收盘越靠近最高价该值越大，衡量收盘强度。
KSFT2(2*$close-$high-$low)/($high-$low+1e-12)收盘位置偏移占振幅的比例（标准化到 [-1,1] 附近）。

2. 当日价格因子（price，4 个）——归一化后的当日价

默认只取当日（window=0）的 OPEN/HIGH/LOW/VWAP，都除以 $close 归一化（当日 close 归一后恒为 1，故不单列 CLOSE0）。

名称表达式含义 / 计算
OPEN0$open/$close当日开盘价 / 收盘价，反映开盘相对收盘的高低。
HIGH0$high/$close当日最高价 / 收盘价。
LOW0$low/$close当日最低价 / 收盘价。
VWAP0$vwap/$close当日成交量加权均价 / 收盘价，反映收盘价在全天成交重心之上还是之下。

3. 滚动统计量因子（rolling，29 类 × 5 窗口 = 145 个）

下表每一类都会在 d ∈ {5,10,20,30,60} 上各生成一个因子（如 MA5, MA10, MA20, MA30, MA60）。表达式中的 d 代表窗口天数。

名称表达式（以窗口 d 为例）含义 / 计算
ROC{d}Ref($close,d)/$close变动率（Rate of Change）：d 天前收盘价与今日之比，反映 d 日动量/涨跌。
MA{d}Mean($close,d)/$close简单移动均线：过去 d 日收盘均价，归一化。<1 表示今日价高于均线（偏强）。
STD{d}Std($close,d)/$close过去 d 日收盘价标准差，归一化——衡量近期波动率。
BETA{d}Slope($close,d)/$close过去 d 日收盘价对时间做线性回归的斜率，归一化——价格趋势的陡峭程度。
RSQR{d}Rsquare($close,d)上述线性回归的 R²，衡量趋势的「直线程度」（越接近 1 越像一条直线）。
RESI{d}Resi($close,d)/$close线性回归残差，归一化——价格偏离趋势线的程度。
MAX{d}Max($high,d)/$close过去 d 日最高价，归一化——上方阻力位。
MIN{d}Min($low,d)/$close过去 d 日最低价，归一化——下方支撑位。
QTLU{d}Quantile($close,d,0.8)/$close过去 d 日收盘价的 80% 分位数，归一化——偏高价位水平。
QTLD{d}Quantile($close,d,0.2)/$close过去 d 日收盘价的 20% 分位数，归一化——偏低价位水平。
RANK{d}Rank($close,d)今日收盘价在过去 d 日中的百分位排名，反映当前价格处于近期高位还是低位。
RSV{d}($close-Min($low,d))/(Max($high,d)-Min($low,d)+1e-12)未成熟随机值（KDJ 的 RSV）：收盘价在过去 d 日高低区间中的相对位置，0~1。
IMAX{d}IdxMax($high,d)/d距上一次最高价出现过了几天（占 d 的比例）——Aroon 指标组件，衡量创新高的近期程度。
IMIN{d}IdxMin($low,d)/d距上一次最低价出现过了几天（占 d 的比例）——Aroon 指标组件。
IMXD{d}(IdxMax($high,d)-IdxMin($low,d))/d最高价日与最低价日的时间差（占 d 的比例），正值大暗示近期先高后低（下行动量）。
CORR{d}Corr($close, Log($volume+1), d)过去 d 日「收盘价」与「对数成交量」的相关系数——量价配合度（价涨量增为正相关）。
CORD{d}Corr($close/Ref($close,1), Log($volume/Ref($volume,1)+1), d)过去 d 日「价格变动率」与「成交量变动率」的相关系数——量价变化的同步性。
CNTP{d}Mean($close>Ref($close,1), d)过去 d 日中上涨天数的占比（涨的比例）。
CNTN{d}Mean($close<Ref($close,1), d)过去 d 日中下跌天数的占比。
CNTD{d}CNTP{d} - CNTN{d}上涨天数占比减下跌天数占比——净上涨倾向。
SUMP{d}Sum(Greater($close-Ref($close,1),0),d)/(Sum(Abs($close-Ref($close,1)),d)+1e-12)过去 d 日「总涨幅 / 总绝对涨跌幅」——类似 RSI，衡量上涨力度占比。
SUMN{d}Sum(Greater(Ref($close,1)-$close,0),d)/(Sum(Abs($close-Ref($close,1)),d)+1e-12)过去 d 日「总跌幅 / 总绝对涨跌幅」，理论上 SUMN = 1 - SUMP。
SUMD{d}(总涨幅 - 总跌幅)/(总绝对涨跌幅+1e-12)涨跌力度之差的比值（SUMP - SUMN），近似 RSI 的方向强度。
VMA{d}Mean($volume,d)/($volume+1e-12)成交量移动均线：过去 d 日均量 / 今日量，>1 表示今日缩量。
VSTD{d}Std($volume,d)/($volume+1e-12)过去 d 日成交量标准差，归一化——量能波动性。
WVMA{d}Std(Abs($close/Ref($close,1)-1)*$volume,d)/(Mean(同前,d)+1e-12)成交量加权的价格波动率：以「涨跌幅×成交量」的变异系数衡量放量波动。
VSUMP{d}Sum(Greater($volume-Ref($volume,1),0),d)/(Sum(Abs($volume-Ref($volume,1)),d)+1e-12)过去 d 日「总放量 / 总绝对变量」——成交量版的 SUMP。
VSUMN{d}Sum(Greater(Ref($volume,1)-$volume,0),d)/(...+1e-12)过去 d 日「总缩量 / 总绝对变量」，VSUMN = 1 - VSUMP。
VSUMD{d}(总放量 - 总缩量)/(总绝对变量+1e-12)放量与缩量之差的比值——成交量版 RSI 方向强度。
数一数29 类 × 5 窗口 = 145，加上 9 个 kbar + 4 个 price = 158，因子集因此得名。你也可以在 handler config 里通过 rolling.include/exclude 只启用部分算子，因子数量随之变化。

二、Alpha360（360 个「原始行情」因子）

Alpha360 的哲学与 Alpha158 相反：不做人工特征工程，而是把过去 60 天的 6 类价量原始序列摊平成 360 列，交给模型（尤其是 GRU/LSTM/Transformer 等时序深度模型）自己提取特征。

6 类字段 × 60 天 = 360。每类都除以当日 $close（成交量除以当日 $volume）归一化：所以当日值恒为 1（CLOSE0=1, VOLUME0=1），越往前的天数是相对今日的比值。命名里的数字是「几天前」——CLOSE59 是 59 天前，CLOSE0 是今天。

字段族列名范围表达式（第 i 天前，i=1..59；i=0 为当日）含义 / 计算
收盘价CLOSE0 … CLOSE59Ref($close,i)/$close（当日 $close/$close=1）过去 60 日收盘价序列，相对今日收盘归一化。反映价格走势形态。
开盘价OPEN0 … OPEN59Ref($open,i)/$close过去 60 日开盘价序列 / 今日收盘价。
最高价HIGH0 … HIGH59Ref($high,i)/$close过去 60 日最高价序列 / 今日收盘价。
最低价LOW0 … LOW59Ref($low,i)/$close过去 60 日最低价序列 / 今日收盘价。
成交均价VWAP0 … VWAP59Ref($vwap,i)/$close过去 60 日成交量加权均价序列 / 今日收盘价。
成交量VOLUME0 … VOLUME59Ref($volume,i)/($volume+1e-12)过去 60 日成交量序列 / 今日成交量。反映量能变化形态。
为什么 Alpha360 更适合深度模型它给的是近乎原始的时间序列，没有替你算好均线、RSI 这些「答案」。树模型难以从裸序列里自动发现时序模式，而 RNN/Transformer 天生擅长；反过来，把 Alpha360 喂给 LightGBM 通常不如 Alpha158。这就是「特征工程 vs 表示学习」的取舍。

三、两套因子集的标签（label）

因子是「输入 X」，标签是「要预测的 Y」。两套默认标签相同（handler.py:89 / 151）：

名称表达式含义
LABEL0Ref($close,-2)/Ref($close,-1) - 1次日到后日的收益率：-1=明天、-2=后天。即「明天按收盘买入、后天按收盘卖出」的收益。用未来价格（负 Ref）构造，仅出现在训练标签里，绝不会进入特征，因此不构成未来函数。
变体Alpha158vwap / Alpha360vwap 把标签换成用 $vwap 计算（Ref($vwap,-2)/Ref($vwap,-1)-1），更贴近「均价成交」的现实假设。
