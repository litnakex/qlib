CHAPTER 02

环境搭建与数据准备

工欲善其事，必先利其器。本章带你把 Qlib 在本机装好、把中国 A 股数据下好，并成功执行第一行 qlib.init()——为后续所有实战打下地基。

1. 为什么不能只用 pip install

很多人第一次接触 Qlib，习惯性地敲下 pip install pyqlib 或 pip install -e .，然后在计算滚动因子时莫名其妙地报错或者性能奇慢。原因藏在一个容易被忽略的细节里：Qlib 的核心计算依赖两个用 Cython 写的 C++ 扩展。

打个比方：Qlib 的 Python 代码像一份「宜家家具的说明书」，而这两个 Cython 文件是「还没组装的木板」。pip install 只把说明书放进了你家，但木板（.pyx 源文件）还需要一台电锯（C++ 编译器）现场加工成能用的家具（.so 动态库）。这一步就是编译。

这两个扩展位于 qlib/data/_libs/：

qlib/data/_libs/
├── __init__.py
├── rolling.pyx      <- 滚动窗口算子 (Rolling)，如 Ref/Mean/Std
└── expanding.pyx    <- 扩展窗口算子 (Expanding)，如累计统计

它们负责 Qlib 表达式引擎里最热的滚动/扩展窗口计算。setup.py:9-24 明确把它们声明为需要编译的 Extension，语言是 c++。而 pyproject.toml 的构建系统虽然列了 cython/numpy 依赖，但普通的 editable 安装并不会稳妥地触发这两个扩展的原地编译，因此仓库额外提供了 Makefile 来兜底。

提示判断扩展是否编译成功，最简单的办法是看 qlib/data/_libs/ 目录下有没有出现 rolling*.so 和 expanding*.so（Windows 上是 .pyd）。有则说明「木板已组装」。

2. 用 Makefile 正确安装

Qlib 仓库根目录的 Makefile 把安装流程封装成了几个语义清晰的目标（target）。核心是理解 prerequisite 这个「先决步骤」——它专门负责编译上面那两个 Cython 扩展。

看 Makefile:61-69 的逻辑（简化版）：如果目录里已存在 .so 就跳过，否则先装 cython numpy，再调用 setuptools 把 rolling.pyx / expanding.pyx 就地（--inplace）编译成动态库。

# Makefile 中的关键目标组合
install: prerequisite dependencies   # 编译扩展 + 以 editable 方式安装核心
dev:     prerequisite all            # 编译扩展 + 安装全部可选依赖组

因此在你 git clone 仓库后，推荐这样安装（先建议用 conda 建一个干净环境）：

# 1) 建一个隔离环境（Python 3.8~3.12 均可，见 pyproject.toml classifiers）
conda create -n qlib python=3.10 -y
conda activate qlib

# 2) 进入仓库根目录
cd /path/to/qlib

# 3) 只装核心（含扩展编译）——日常研究足够
make install

# 或：连同 dev/lint/docs/test/analysis/rl 等全部可选依赖一起装
make dev
陷阱不要只跑 pip install -e . 就以为装好了。它对应的只是 Makefile 里的 dependencies 目标（Makefile:76-77），缺了 prerequisite 编译步骤。若你坚持用 pip，请务必先手动执行 make prerequisite 或运行 python setup.py build_ext --inplace 编译扩展。

如果没有 make（例如某些 Windows 环境），可以手动复刻 prerequisite 的核心动作：

pip install --upgrade setuptools wheel cython numpy
python setup.py build_ext --inplace   # 生成 rolling.so / expanding.so
pip install -e .

3. 看懂可选依赖组

Qlib 把「非人人都需要」的依赖拆成了若干可选组，定义在 pyproject.toml:59-108 的 [project.optional-dependencies] 里。安装时用中括号选装，例如 pip install -e .[rl]。下面是各组用途速查：

组名包含的关键依赖什么时候需要
devpytest, statsmodels开发/跑单元测试
rltorch, tianshou<=0.4.10, numpy<2.0.0强化学习模块（如 order execution）
lintblack, pylint, flake8, mypy代码风格与静态检查
docssphinx, sphinx_rtd_theme本地构建文档
testyahooquery, baostockPIT/数据抓取相关测试
analysisplotly, statsmodels交互式绘图与分析报告
clientpython-socketio<6, tables连接远程数据服务（client 模式）
重点：torch 不是核心依赖注意 dependencies（pyproject.toml:27-57）里没有 torch。深度学习模型属于 rl 组或需你自行安装。更要命的是 rl 组会把 numpy 锁死在 <2.0.0（pyproject.toml:70）——因为 macOS 上部分 PyTorch 版本尚不完全支持 numpy 2.x。如果你既要跑 GBDT 又要跑神经网络，装 rl 组后就别再随手把 numpy 升级到 2.x 了，否则可能触发 ABI 不兼容的诡异崩溃。
建议只想快速上手 LightGBM/Alpha158 基线？lightgbm 已在核心依赖里，直接 make install 即可，无需任何可选组。等要玩深度模型时再 pip install -e .[rl]。

4. 下载数据

装好代码只是一半，Qlib 还需要「粮食」——历史行情数据。官方提供了打包好的中国 A 股日频数据，用 scripts/get_data.py 一键下载。这个脚本本质上是用 fire 把 qlib.tests.data.GetData 类暴露成命令行（见 scripts/get_data.py:4-8）。

# 下载精简版中国 A 股日线数据（体积小，适合入门）
python scripts/get_data.py qlib_data \
    --name qlib_data_simple \
    --target_dir ~/.qlib/qlib_data/cn_data \
    --interval 1d \
    --region cn

参数逐一对照 GetData.qlib_data 的签名（qlib/tests/data.py:153-161）：

  • --name qlib_data_simple：数据集名称。可选 qlib_data（全量）或 qlib_data_simple（精简版，下载更快）。
  • --target_dir ~/.qlib/qlib_data/cn_data：数据落盘目录，也是后面 init 的 provider_uri。
  • --interval 1d：频率，日频。也支持 1min 分钟线（目录名通常用 cn_data_1min）。
  • --region cn：区域，中国 A 股。美股用 us。
~/.qlib/qlib_data/cn_data/     <- 这就是 provider_uri
├── calendars/                 <- 交易日历
├── instruments/               <- 股票池 (all / csi300 / csi500 ...)
└── features/                  <- 每只股票的 $close/$volume 等二进制特征
注意qlib_data 默认会先删除 target_dir 下已存在的旧数据（delete_old=True），再解压新数据。若你担心误删，可加 --exists_skip True 让已存在时跳过下载（qlib/tests/data.py:194）。数据默认放在 ~/.qlib/qlib_data/ 下，记住这个约定路径，几乎所有教程与示例配置都以它为默认 provider_uri。

5. 初始化 qlib.init()

数据到位后，任何 Qlib 的数据/模型 API 都必须在一次成功的 qlib.init() 之后才能使用。可以把 init 理解成「开机」：它读取配置、注册 provider、把 provider_uri 指向的数据目录挂上，之后 D.features、模型训练、回测才认得数据在哪。

import qlib
from qlib.constant import REG_CN

qlib.init(
    provider_uri="~/.qlib/qlib_data/cn_data",  # 就是刚才 target_dir
    region=REG_CN,                             # "cn"，见 qlib/constant.py:10
)

init 定义在 qlib/__init__.py:25。它会遍历 provider_uri，对本地路径检查是否存在（不存在会打印 error 提示，qlib/__init__.py:65-72），最后调用 C.register() 完成注册，并打印 qlib successfully initialized。

除了直接调 init，源码里还提供了两个更省心的入口：

  • init_from_yaml_conf(conf_path)（qlib/__init__.py:188）：从一个 YAML 配置文件读取参数再 init。工作流（qrun）走的就是这条路。
  • auto_init()（qlib/__init__.py:243）：自动向上查找项目里的 config.yaml 并据此初始化；找不到就用默认配置。它默认带 skip_if_reg=True，即「已初始化过就跳过」，方便在脚本被多次导入时避免重复注册导致的 recorder 丢失。
提示init 里的 skip_if_reg 参数（qlib/__init__.py:47-52）很实用：在一次实验 R.start 运行途中若不小心再次 init，会丢失当前 recorder；设为 True 可跳过重复初始化保护现场。

6. macOS 上的 OpenMP 坑

在 macOS（尤其 Apple Silicon）上，LightGBM、numpy/scipy 的底层 OpenMP 库有时会因多个运行时（如 Homebrew 的 libomp 与 conda 自带的）同时加载而冲突，表现为 Python 进程直接 Segmentation fault 闪退，且没有可读的 Python 堆栈——非常迷惑。

最稳妥的规避手段是把 OpenMP 线程数强制设为 1，让它退化成单线程避免竞态：

# 在运行 Python 之前，于 shell 里导出环境变量
export OMP_NUM_THREADS=1
export MKL_NUM_THREADS=1
export KMP_DUPLICATE_LIB_OK=TRUE   # 允许重复加载 libomp（临时缓解）

python your_script.py

也可以在 Python 代码最顶部（务必在 import numpy / lightgbm 之前）设置：

import os
os.environ["OMP_NUM_THREADS"] = "1"
os.environ["KMP_DUPLICATE_LIB_OK"] = "TRUE"

import qlib  # 之后再导入
dangerKMP_DUPLICATE_LIB_OK=TRUE 只是「让程序别崩」的应急开关，可能掩盖真正的库冲突。长期方案是保证环境里只有一份 libomp（比如统一用 conda 装 llvm-openmp，避免和 Homebrew 混装）。

7. 装好后的验证：最小可运行片段

把前面的步骤串起来，用下面这段「体检代码」确认环境、数据、扩展三者都就绪。能打印出 K 线数据，就说明一切正常。

import os
os.environ.setdefault("OMP_NUM_THREADS", "1")  # macOS 保险

import qlib
from qlib.constant import REG_CN
from qlib.data import D

# 1) 初始化
qlib.init(provider_uri="~/.qlib/qlib_data/cn_data", region=REG_CN)

# 2) 看看交易日历（验证 calendars 目录）
cal = D.calendar(start_time="2020-01-01", end_time="2020-01-10", freq="day")
print("最近交易日:", cal[-3:])

# 3) 取一只股票的行情特征（验证 features 目录 + Cython 扩展）
df = D.features(
    instruments=["SH600000"],
    fields=["$close", "$volume", "Ref($close, 1)", "Mean($close, 5)"],
    start_time="2020-01-01",
    end_time="2020-01-10",
    freq="day",
)
print(df)

其中 Ref 与 Mean 正是走 rolling.pyx 编译出的算子——如果这一步没报 ImportError 并正常出结果，说明 Cython 扩展编译成功。D.features 的签名见 qlib/data/data.py:1162，D.calendar 见 qlib/data/data.py:71。

成功标志控制台先出现一行 qlib successfully initialized based on client settings.，随后打印出一张带 $close / $volume 列的 DataFrame，即大功告成，可以进入第 3 章学习数据层了。

📝 课后练习题

1.（概念题）为什么单纯 pip install -e . 安装 Qlib 后可能出问题？正确的安装命令是什么？
参考答案
因为 Qlib 依赖两个 Cython 编写的 C++ 扩展 rolling.pyx 和 expanding.pyx（位于 qlib/data/_libs/），它们承担滚动/扩展窗口计算。pip install -e . 只对应 Makefile 的 dependencies 目标，缺少编译扩展的 prerequisite 步骤。正确做法是 make install（= prerequisite + dependencies）或 make dev；无 make 时先 python setup.py build_ext --inplace 再 pip install -e .。
2.（概念题）rl 可选依赖组会对 numpy 版本做什么限制？为什么？
参考答案
见 pyproject.toml:67-71，rl 组把 numpy 锁在 <2.0.0，并安装 torch 与 tianshou<=0.4.10。原因是在 macOS 上部分 PyTorch 版本不完全支持 numpy 2.x，为避免 ABI 冲突而降版。此外 torch 并不在核心 dependencies 里，是可选依赖。
3.（代码填空）补全下载精简版 A 股日频数据的命令。 
python scripts/get_data.py qlib_data \
    --name ______ \
    --target_dir ______ \
    --interval ______ \
    --region ______
参考答案
--name qlib_data_simple、--target_dir ~/.qlib/qlib_data/cn_data、--interval 1d、--region cn。数据默认落在 ~/.qlib/qlib_data/，其中 target_dir 即后续 init 的 provider_uri。
4.（改错题）下面的代码在 qlib.init() 之前就调用了数据 API，会出问题。请指出并改正。 
from qlib.data import D
df = D.features(["SH600000"], ["$close"])
import qlib
qlib.init(provider_uri="~/.qlib/qlib_data/cn_data")
参考答案
任何数据/模型 API 都必须在成功 qlib.init() 之后调用。应先 import qlib 并 qlib.init(provider_uri=..., region=REG_CN)，再调用 D.features(...)。此外最好显式传入 region=REG_CN。
5.（思考题）在 macOS 上运行时进程直接 Segmentation fault 且没有 Python 堆栈，你会怎么排查与缓解？
参考答案
大概率是 OpenMP 库（libomp）被多个运行时重复加载引发的冲突。应急手段：在启动前设 OMP_NUM_THREADS=1、MKL_NUM_THREADS=1，必要时 KMP_DUPLICATE_LIB_OK=TRUE（在 import numpy/lightgbm 之前设置）。根治方案是保证环境只存在一份 libomp，例如统一用 conda 的 llvm-openmp，避免与 Homebrew 混装。
6.（思考题）auto_init() 与直接 init() 有何区别？它默认的 skip_if_reg=True 解决了什么问题？
参考答案
auto_init()（qlib/__init__.py:243）会向上查找项目里的 config.yaml 并据此初始化，找不到才退回默认 init，适合项目化脚本免手写路径。它默认 skip_if_reg=True：若 Qlib 已注册过则跳过重复初始化，避免在实验 R.start 运行中被再次 init 而丢失当前 recorder。
